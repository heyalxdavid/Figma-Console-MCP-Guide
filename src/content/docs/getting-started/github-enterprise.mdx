If your company uses a GitHub Enterprise instance (like `github.intuit.com`), pushing code from Cursor requires a few extra steps compared to regular `github.com`. This guide walks through installing the GitHub CLI, authenticating via SSO, and creating your first repo.

## Prerequisites

- [Cursor installed](../cursor-setup/)
- A GitHub Enterprise account with SSO access
- [Homebrew](https://brew.sh/) installed (macOS)
- An SSH key on your machine (most Macs have one at `~/.ssh/id_ed25519.pub`)

## Installing the GitHub CLI

The `gh` CLI makes it simple to create repos, manage pull requests, and authenticate without leaving the terminal.

<Steps>

1. **Install with Homebrew**

   Open the terminal in Cursor (`Ctrl + ~`) and run:

   ```bash
   brew install gh
   ```

2. **Verify the installation**

   ```bash
   gh --version
   ```

   You should see a version number like `gh version 2.89.0`.

</Steps>

## Authenticating with GitHub Enterprise

Standard `gh auth login` defaults to `github.com`. For an enterprise instance, you need to tell it your hostname.

<Steps>

1. **Start the login flow**

   ```bash
   gh auth login
   ```

2. **Select your account type**

   When prompted "Where do you use GitHub?", choose **Other** and enter your enterprise hostname, for example `github.intuit.com`.

   <Aside type="caution">
   Double-check the hostname for typos. Entering `ithub.intuit.com` instead of `github.intuit.com` will fail with a connection error.
   </Aside>

3. **Choose SSH as your protocol**

   Select **SSH** when asked for your preferred protocol for Git operations.

4. **Upload your SSH key**

   The CLI will detect your existing SSH key (usually at `~/.ssh/id_ed25519.pub`). Select it and give it a title like "Macbook" or your machine name.

5. **Authenticate in the browser**

   Choose **Login with a web browser**. This opens your company's SSO page where you sign in with your enterprise credentials.

6. **Verify authentication**

   ```bash
   gh auth status
   ```

   You should see output confirming your account and active session:

   ```
   github.intuit.com
     ✓ Logged in to github.intuit.com account yourname (keyring)
     - Active account: true
     - Git operations protocol: ssh
   ```

</Steps>

## Initializing Git and Pushing to a New Repo

Once authenticated, you can create a repo and push your project in a few commands.

<Steps>

1. **Initialize git in your project**

   ```bash
   git init
   git add -A
   ```

2. **Rename the default branch to `main`**

   ```bash
   git branch -m main
   ```

3. **Make your initial commit**

   ```bash
   git commit -m "Initial commit"
   ```

4. **Create the repo on GitHub Enterprise and push**

   The `GH_HOST` environment variable tells the CLI which GitHub instance to target:

   ```bash
   GH_HOST=github.intuit.com gh repo create your-repo-name \
     --public \
     --description "Your project description" \
     --source . \
     --push
   ```

   This creates the repo under your account, sets up the `origin` remote, and pushes your code in one step.

5. **Verify on GitHub**

   Visit `https://github.intuit.com/yourname/your-repo-name` to confirm everything landed.

</Steps>

## Enterprise Workflow Validation

Many enterprise GitHub instances enforce rules on CI workflow files. For example, Intuit's GitHub requires a `validation` job in every workflow that references a specific reusable workflow.

If your push is rejected with a message like:

```
The IBP 'validation' job is required in this workflow.
```

Add the required validation job to your `.github/workflows/*.yml` files:

```yaml
jobs:
  validation:
    uses: actions-intuit/ibp-wf/.github/workflows/validation.yml@v1

  build:
    needs: validation
    runs-on: ubuntu-latest
    steps:
      # your build steps...
```

Key points:

- The `validation` job must be present in every workflow file
- All other jobs must include `needs: validation`
- The `uses` field must point to the exact reusable workflow your org requires

<Aside>
Check with your platform team for the specific reusable workflow URL your organization requires. The example above is specific to Intuit's setup.
</Aside>

## Troubleshooting

- **"error connecting to hostname"**: Check for typos in the hostname. Run `gh auth login` again with the correct enterprise URL.
- **"Bad credentials" (HTTP 401)**: Your token may be expired or revoked. Re-authenticate with `gh auth login`.
- **"pre-receive hook declined"**: Your enterprise has push rules. Read the error message carefully; it usually tells you exactly what's missing from your workflow files.
- **SSH key not found**: Generate one with `ssh-keygen -t ed25519` and try the login flow again.
